# Privacy and local data

Converge Mini is designed so its first-party Worker does not receive or store private message content.

## Data handled in the browser

- Farcaster host context such as display name, username, avatar URL, and FID is used only as an unverified display hint. A backend identity request instead uses the FID from a verified Farcaster Quick Auth token.
- The Farcaster host account signs routine XMTP setup and messaging access. For an explicit one-time binding to a separate ENS inbox, the exact ENS-resolved owner signs through an external WalletConnect wallet to authorize the target installation, then the Farcaster wallet signs its identity reassignment into that inbox. The app does not generate, receive, export, or store either wallet's private key.
- The active wallet address, XMTP inbox/installation identifiers, conversations, messages, drafts, and consent state are processed in the browser.
- After XMTP is ready, the app may request a short-lived Quick Auth token for optional first-party identity calls. The app sends it only in the authorization header and does not persist it in D1 or browser application storage.
- Farcaster host context may expose whether the Mini App is added and a client-visible notification URL/token. That context is only an untrusted display hint: the browser never sends it to the first-party API, stores it, logs it, or passes it through application state. Only a separately signed and current-app-key-verified server webhook may establish notification permission.
- After a successful **No thanks**, the browser stores one non-authoritative dismissal bit keyed by the host-context FID. It contains no name, address, or XMTP identifier and only prevents a repeat background Quick Auth request on that browser; D1 remains the trusted account-wide preference.
- A returned ENS name/address candidate is compared read-only with the active XMTP signer and inbox. The app can determine that the address is active, in the same inbox, in a different inbox, or has no inbox; this check does not add, remove, merge, or migrate an XMTP identity.
- After XMTP confirms the binding, the browser stores only the normalized public ENS name, checksummed target/source addresses, expected target inbox ID, and Farcaster signer kind/chain in `localStorage`, namespaced by the host-context FID. It never puts the pairing URI, WalletConnect topic, signature, key, conversation, or message in this record. WalletConnect is disconnected after the one-time action. On every later launch, Farcaster must expose the exact saved source wallet with the matching signer kind; a smart-contract wallet must also remain on its saved chain. XMTP must resolve that source to the expected target inbox before any messages render. Corrupt, unreadable, or obsolete pre-binding selectors require explicit recovery instead of being treated as proof of a completed association.
- When the user explicitly submits a dot-separated ENS recipient, the browser sends that name in the JSON body of a protected first-party request. The Worker transiently processes the verified caller FID and recipient query together, which reveals recipient-search relationship metadata even though application code does not persist or log either value. The response contains only the ENSIP-15-normalized name and checksummed public address, or an explicit no-record result.
- The browser sends already-public peer Ethereum addresses, in batches of at most 12, to the protected first-party participant-identity route. This means the first-party Worker transiently processes the verified caller FID and those peer addresses in the same request, revealing caller-to-peer relationship metadata even though it is not persisted or logged by application code. Positive display metadata stays in memory for at most ten minutes, complete negative results for two minutes, and partial failures retry after one minute. The response contains the address and optional registered-fname, ENS, and Basename metadata; it does not return the intermediate Farcaster FID.
- XMTP's Browser SDK persists its database in origin-private file-system storage under `miniapp.converge.cv`. The database is local but is not encrypted at rest by this app. Anyone who can access the unlocked browser profile or exploit same-origin script execution may be able to read decrypted data.
- A same-origin service worker caches only the public HTML shell and static scripts, styles, fonts, and images needed to render it after an online visit. It retains at most the current and previous complete fingerprinted shell generations so an update or rollback has one known-good fallback without unboundedly competing with XMTP OPFS for origin quota. It does not cache `/api/*`, `/.well-known/*`, Quick Auth responses, XMTP traffic, conversation previews, messages, drafts, attachments, or notification data. Cache Storage is not a second message store.
- Each XMTP inbox uses its own Browser SDK database. Binding and reopening the target does not copy messages into `localStorage`, merge histories, delete the prior database, or revoke either installation. Cached message content remains in XMTP's OPFS database; Web Storage holds only the small public binding-label record and acknowledgement/preference hints described above.
- A Web Lock prevents two same-origin tabs/windows from opening the XMTP database concurrently.
- The app asks the browser for persistent storage after a user opens the inbox. If the browser grants only best-effort storage, the inbox remains usable but shows a dismissible warning that local history can be evicted under storage pressure. Dismissing it stores one local acknowledgement bit with no account, wallet, or message identifier; clearing site data restores the warning. The complete storage and recovery disclosure remains available in the identity/privacy menu.

Clearing site data for `miniapp.converge.cv` deletes the local databases, static offline cache, saved ENS-binding label, any WalletConnect library state, and other app storage, but it does not delete messages from XMTP, undo the Farcaster identity reassignment, or revoke an XMTP installation. The next launch still follows the Farcaster identity's network association, even if the local ENS label was lost. Clearing storage can create another installation on the next setup, so the app never revokes other installations automatically.

## Data sent to other services

- The Farcaster host supplies display context and the EIP-1193 wallet provider.
- For an explicit ENS identity binding, Reown/WalletConnect services receive the public project ID and ordinary connection metadata needed for the one-time dapp session; the relay carries encrypted session traffic, the WalletConnect RPC service may answer public chain reads such as account bytecode, and WalletConnect Verify may check the requesting origin. The pairing URI is shown locally as a QR/raw URI or handed to MetaMask, and application code does not log or send it to the first-party Worker. Routine later XMTP sessions do not initialize WalletConnect.
- Farcaster Quick Auth issues the token used for optional protected identity calls. If no in-memory token exists, the host may request a Farcaster sign-in approval after XMTP is already usable. The Worker verifies the token's signature, issuer, expiry, exact domain/audience, and positive FID subject against Farcaster's published keys.
- When notification enrollment is promoted, the Farcaster client sends signed add/enable/disable/remove events directly to the first-party webhook. The Worker asks the configured Hub provider whether the signing client app key is currently valid. The provider necessarily receives the public client FID and app-key lookup plus ordinary network metadata; it does not receive the notification token, delivery URL, XMTP identifiers, or message content from application code.
- For an authenticated ENS lookup, the Worker sends the verified FID to Farcaster's official primary-address endpoint and asks configured public Ethereum RPC providers to resolve the returned public address's ENS primary name and then resolve that name back to the same address. Those services necessarily receive ordinary network metadata and the public identifier being queried.
- For an explicitly submitted ENS recipient, configured public Ethereum RPC providers and, when CCIP Read is required, the configured ENS gateway receive the normalized public name for mainnet forward resolution. The query is sent in a protected POST body to the first-party Worker rather than a URL, but those resolver services still receive ordinary network metadata and the public name.
- For a participant-label lookup, configured Ethereum RPC providers receive the public peer addresses for ENS and ENSIP-19 Basename reverse resolution. When the optional Base RPC secret is configured, that provider receives one bounded batch for a read against Neynar's experimental Verifications contract, and the official FName Registry receives any positive FID needed to fetch its current registered fname. That contract can collapse an address shared by multiple FIDs, excludes custody-only mappings, and a registered fname need not be the profile's selected username, so this value appears only as an explicitly secondary registry hint beside resolver-backed wallet metadata—not authentication or a canonical account link.
- XMTP network and payer-Gateway services process protocol traffic and messaging metadata needed to deliver end-to-end encrypted messages.
- On a newly registered Mini App installation, Converge Mini sends XMTP's best-effort history-sync request. A compatible existing installation must be online to re-encrypt and upload an archive for recovery; the archive service is distinct from same-origin OPFS persistence. Sending the request does not prove that an archive has finished importing, so later manual or foreground refreshes can continue to surface recovered history.
- Avatar/image URLs supplied by the Farcaster host can cause the browser to request an external image host.
- Cloudflare necessarily handles ordinary HTTP request metadata for serving the app and sampled Worker observability. Application code adds no message, wallet, FID, inbox, conversation, signature, draft, or token fields to logs.

XMTP protects message content end to end; it does not make wallet identity, participants, timing, IP/network access, or all delivery metadata anonymous.

## First-party backend inventory

The Worker has two purpose-limited D1 tables. `ens_identity_preferences` contains only:

- the Quick Auth-verified Farcaster FID as the primary key;
- `choice`, constrained to `accepted` or `dismissed`; and
- an `updated_at` Unix timestamp.

`farcaster_notification_subscriptions` is an unadvertised foundation until a current-Hub verifier credential is configured and the manifest is deliberately promoted. Each row contains only:

- the signed webhook's verified Farcaster user FID and verified client app FID as the composite primary key;
- one AES-256-GCM ciphertext containing the exact notification delivery URL and token together;
- a fresh nonce, encryption-key version, and created/updated Unix timestamps.

The encryption additional data binds the ciphertext to both FIDs and the canonical domain, so moving ciphertext between rows or origins fails authentication. The exact plaintext URL and token exist only transiently while processing a verified webhook or future delivery. A verified disable, removal, or add event without details deletes the exact client row. Authenticated `DELETE /api/me` deletes all rows for the verified user FID. Farcaster events currently have no timestamp or sequence, so a future delivery queue must recheck the active row and treat deletion as authoritative instead of assuming total event ordering.

The Worker does not store the ENS recipient query, ENS binding metadata, ENS name, Basename, fname, peer wallet address, XMTP inbox or installation ID, Quick Auth token, Farcaster profile, app key, webhook signature, conversation, message, or draft. `POST /api/resolve` forward-resolves one recipient name statelessly, and `POST /api/identities` resolves public participant labels statelessly. `GET /api/me/ens` resolves the public candidate again on each request. `PUT /api/me/ens-preference` replaces the choice for that FID, and authenticated `DELETE /api/me` deletes the ENS row plus encrypted notification rows. The ENS choice remains until it is replaced or deleted. **Delete saved ENS choice** in the identity/privacy menu invokes that protected deletion route; deleting restores the unset state, so the safe optional offer can appear again. The separate ENS binding-label record never leaves the browser through a first-party application request.

Production and preview use separate D1 databases through the `PREFERENCES` binding. There is no analytics SDK, advertising tracker, contacts database, identity-link table, plaintext notification token store, or plaintext-message API. The configured ownership values are Farcaster's public account-association strings.

Application code does not log the raw ENS recipient query, Quick Auth token, FID, resolved address/name, saved preference, notification URL/token, Farcaster Signature, or XMTP identifiers. The notification table does not make closed-app XMTP alerts operational: that requires a separately privacy-reviewed persistent observer and delivery queue before the UI may promise incoming-message alerts.
